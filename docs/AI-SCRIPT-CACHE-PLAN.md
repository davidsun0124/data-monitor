# AI 生成脚本缓存方案（自适应版）

## Context

用户需求演进：
1. **第一遍**：AI 自己跑 → 生成脚本
2. **后期**：直接运行脚本执行（低成本、低延迟）
3. **AI 转型**：从"执行者"变成"监控者" - 监控脚本是否正常运行
4. **AI 介入条件**：脚本过时 / 不满足需要 / 发生异常时，AI 介入调整脚本

## 方案设计

### 核心流程

```
定时触发
    │
    ▼
┌─────────────────┐
│  执行脚本        │ ← 后期常态
└────────┬────────┘
         │ 正常 → 解析结果 → 发送告警（无需 AI）
         │
         ▼ 异常/缺失/过期
┌─────────────────┐
│  调用 AI 介入    │ ← 脚本异常时
│  - 分析问题      │
│  - 调整脚本      │
│  - 重新执行      │
└─────────────────┘
```

### 脚本状态机

| 状态 | 触发条件 | 行为 |
|------|----------|------|
| `MISSING` | 脚本不存在 | 调用 AI 生成 |
| `STALE` | 任务 .md 比脚本新 | 调用 AI 重新生成 |
| `EXEC_ERROR` | 脚本执行失败（DB 连接、语法错误等） | 调用 AI 分析并修复 |
| `RESULT_ABNORMAL` | 脚本执行结果异常（如数据量突增、发现新类型异常） | 调用 AI 分析，决定是否调整 |
| `OK` | 脚本正常执行 | 记录结果，结束 |

### AI 介入时的 Prompt 设计

```markdown
## 背景
- 任务定义：{task_name}.md 的内容
- 脚本路径：generated_scripts/{task_name}.sql
- 异常信息：{reason}（MISSING / EXEC_ERROR / RESULT_ABNORMAL / STALE）
- 上次执行结果：{last_result}

## 要求
1. 如果脚本不存在或过期：根据任务定义生成脚本
2. 如果脚本执行错误：分析错误原因并修复脚本
3. 如果结果异常：分析数据，判断是否需要调整脚本逻辑
4. 将生成的脚本保存到 `generated_scripts/{task_name}.sql`
5. 输出 SUMMARY_JSON 格式的巡检结果
```

### 生成的脚本格式

```sql
-- Auto-generated by AI on {timestamp}
-- Task: {task_name}
-- DO NOT EDIT MANUALLY

SELECT 'CHECK_1' AS check_type, COUNT(*) AS cnt
FROM flying_api.FIM_FixedPrice
WHERE CheckDate = DATE_SUB(CURDATE(), INTERVAL 1 DAY)
  AND ABS(Fee0+Fee2+... - BeforeDiscountFee) > 0.01;

SELECT 'CHECK_2' AS check_type, COUNT(*) AS cnt
FROM flying_api.FIM_FixedPrice
WHERE CheckDate = DATE_SUB(CURDATE(), INTERVAL 1 DAY)
  AND AHS_Size_Fee > 50;

-- 输出为 JSON 行格式，方便解析
{"check":"BILL_CONSISTENCY","anomaly_count":0,"samples":[]}
{"check":"AHS_FEE_HIGH","anomaly_count":5,"samples":["TN001","TN002","TN003","TN004","TN005"]}
```
